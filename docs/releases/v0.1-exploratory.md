## Implemented families

### C2 — Basic Reachability (calibration)

- **Question:** Is target state `q_t` reachable from `q_0`?
- **Certificates:** `trace_witness` (positive), `unreachability_witness` (negative)
- **Generator:** `fsmreasonbench.generator.reachability`
- **Difficulty controls:** `min_witness_length`, `max_witness_length`, `include_negative`, `state_count`

### F1 — Separation / Witness (flagship, first slice)

- **Question:** Are DFA A and B equivalent?
- **Certificates:** `distinguishing_trace` (non-equivalent), `equivalence_witness` (equivalent)
- **Generator:** `fsmreasonbench.generator.separation` (random, constructive_decoy, legacy constructive)
- **Difficulty controls:** state counts, `min_distinguishing_trace_length`, `max_distinguishing_trace_length`, equivalence mix

Every generated item passes **`self_verify_item()`** before persistence: oracle certificate →
independent verifier → pass/fail.

## Evaluation pipeline

End-to-end path for one model response:

```
item (+ answer_key internally)
  → raw_response
  → parse_submission()           # extractability gate
  → verdict vs answer_key.verdict
  → verify_*_certificate()       # independent verifier
  → ScoringRecord + failure_stage
```

### Four independent metrics

| Metric | Field | Meaning |
|--------|-------|---------|
| Extractability | `extractability_rate` | Parseable, schema-valid submission |
| Verdict accuracy | `verdict_accuracy` | Boolean verdict matches gold (when extractable) |
| Certificate validity | `certificate_valid_rate` | Verifier accepts certificate |
| Full correctness | `fully_correct_rate` | Verdict and certificate both correct |

**Failure stages:** `not_extractable`, `verdict_wrong`, `certificate_invalid`, `correct`.

On flagship F1 tasks, **certificate validity is primary**; high verdict accuracy with low
certificate validity is an expected diagnostic pattern, not verified reasoning success.

Transcripts support deterministic **rescore** without re-invoking the model. Baselines
(`oracle`, `random`, `invalid`) exercise the same pipeline with synthetic responses.

Normative protocol: [`docs/specification/evaluation_protocol.md`](../specification/evaluation_protocol.md).

## How to validate cohorts

From the repository root:

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.validate_cohort \
  --cohort-dir cohorts/v0.1-exploratory/c2-reachability-level3

PYTHONPATH=src python3.11 -m fsmreasonbench.cli.validate_cohort \
  --cohort-dir cohorts/v0.1-exploratory/f1-mixed-level3
```

Validation checks:

- Required files (`items.jsonl`, `manifest.json`, `sha256sums.txt`, `README.md`)
- SHA-256 checksums match on-disk files
- Manifest fields and `item_count` consistency
- Per-item line hashes match manifest entries
- Unique `item_id` values in `items.jsonl`
- `self_verify_item()` on every item

Exit code `0` and output `VALID` indicate a intact exploratory cohort. This confirms artifact
integrity for smoke testing; it does **not** certify a public benchmark release.

To seal a new exploratory cohort from generated JSONL:

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.freeze_cohort \
  --items runs/.../items.jsonl \
  --cohort-id <name>-v0.1-exploratory \
  --out-dir cohorts/v0.1-exploratory/<name>
```

`freeze_cohort` rejects duplicate `item_id` values before writing any output.

## How to reproduce capability-surface experiments

Exploratory capability-surface runs are **not** frozen public benchmark results. The workflow
below reproduces the artifact's reporting machinery on locally generated or sealed item batches.

### 1. Baseline capability surface (no LLM)

Generates items per difficulty level and evaluates oracle / random / invalid baselines:

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.run_capability_surface \
  --families C2,F1 \
  --n-per-level 50 \
  --seed 1 \
  --out-dir runs/capability_surface
```

### 2. Multi-model capability surface (Ollama, temperature 0)

Requires a running Ollama instance with the listed models:

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.run_capability_surface_models \
  --models qwen2.5-coder:7b,llama3.1:8b,mistral-nemo:12b,gemma2:9b \
  --levels 1,2,3,4,5 \
  --n-per-level 20 \
  --out-dir runs/capability_surface_models
```

F1 mixed configuration (equivalent + non-equivalent pairs):

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.run_capability_surface_models \
  --families F1 \
  --f1-mixed \
  --models qwen2.5-coder:7b,llama3.1:8b,mistral-nemo:12b,gemma2:9b \
  --levels 1,2,3,4,5 \
  --n-per-level 20 \
  --out-dir runs/capability_surface_models_f1_mixed
```

### 3. Plot and export

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.plot_capability_surface \
  --summary runs/capability_surface_models/combined_summary.json

PYTHONPATH=src python3.11 -m fsmreasonbench.cli.export_capability_surface_report \
  --summary runs/capability_surface_models/combined_summary.json \
  --out-md docs/capability_surface_report.md \
  --out-csv docs/capability_surface_summary.csv
```

### 4. Failure analysis (optional)

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.failure_taxonomy_batch \
  --root runs/capability_surface_models_f1_mixed \
  --out docs/f1_mixed_failure_taxonomy.json
```

**Reproducibility notes:**

- Item generation and scoring are deterministic given seeds and generator code.
- LLM responses are **not** deterministic; archive `results.jsonl` / transcripts to replay scores.
- Outputs under `runs/` are gitignored unless explicitly committed as summaries in `docs/`.
- Committed exploratory summaries (`docs/pilot_v0_*`, `docs/pilot_v1_*`, capability-surface exports) are illustrative reporting examples, not final benchmark claims.

Using a **frozen exploratory cohort** as input (instead of on-demand generation):

```bash
PYTHONPATH=src python3.11 -m fsmreasonbench.cli.run_ollama_batch \
  --model qwen2.5-coder:7b \
  --items cohorts/v0.1-exploratory/c2-reachability-level3/items.jsonl \
  --out runs/ollama_c2_cohort.jsonl \
  --temperature 0

PYTHONPATH=src python3.11 -m fsmreasonbench.cli.summarize_scores \
  --scores runs/ollama_c2_cohort/scores.jsonl
```

## Known limitations

- **Partial family coverage:** only C2 and F1 are end-to-end; F2–F4 and C1 are specified but not wired.
- **Exploratory cohort size:** committed frozen snapshots are small (20 items each) for smoke testing, not representative benchmark panels.
- **Generator bias:** constructive F1 modes may introduce structural regularities; use `audit_f1_items` before structural claims.
- **Local model scope:** exploratory Ollama runs cover a small open-weight model set; results do not generalize broadly.
- **Verdict overstatement:** high `verdict_accuracy` with low `certificate_valid_rate` is expected on F1 — layered metrics are required.
- **No evaluatee bundle split:** full items (including answer keys) are in exploratory freezes; public release will split bundles.
- **No pinned Zenodo environment:** development installs use editable `pip install -e ".[dev]"`; lockfile pinning awaits `v1.0.0`.
- **License unset:** final license choice (MIT vs Apache-2.0) remains open (PROJECT_STATUS U9).
